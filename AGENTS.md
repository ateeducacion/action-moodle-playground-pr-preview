## What This Project Does

This is a **GitHub Action** that automatically adds a "Preview in Moodle Playground" button to pull requests. When a contributor opens a PR on a Moodle plugin repository, this action generates a link that opens the PR's plugin branch directly inside an ephemeral, in-browser Moodle instance powered by Moodle Playground (PHP-WASM).

The action does **not** build, test, or deploy anything. It only:

1. Reads PR metadata (branch, repo, number).
2. Generates a Moodle Playground blueprint JSON that installs the plugin from the PR branch.
3. Encodes the blueprint and builds a Playground URL.
4. Posts the link to the PR (either in the description or as a comment).

## How to Build

```bash
npm install

```

The build compiles `src/index.js` and its `node_modules` dependencies into a single `dist/index.js` file (~1.1 MB). GitHub Actions executes `dist/index.js` directly. **Always rebuild after editing `src/index.js`.**

## Source Code Architecture (`src/index.js`)

The entire action is a single async IIFE. There are no separate modules, no TypeScript, and no test files.

### Execution Flow

```
1. Read inputs (github-token, mode, plugin-path, blueprint, etc.)
2. Resolve PR data:
   - From event payload (pull_request trigger), OR
   - From GitHub API if pr-number input is provided (workflow_run trigger)
3. Extract PR metadata: owner, repo, branch, SHA, etc.
4. Determine blueprint source (priority order):
   a. User-provided `blueprint` JSON string -> validate and use as-is
   b. `plugin-path` -> auto-generate blueprint via buildAutoBlueprint()
   c. `blueprint-url` -> use remote URL directly (no blueprint generation)
5. Build preview URL:
   - blueprint-url mode: ?blueprint-url={encoded-url}
   - blueprint mode: ?blueprint={base64-encoded-json}
6. Build template variables and render button/description/comment
7. Publish to PR:
   - MODE_APPEND: update PR description body with managed HTML-comment block
   - MODE_COMMENT: create or update a bot comment
8. Set action outputs
```

### Auto-Generated Blueprint Format

When `plugin-path` is provided, `buildAutoBlueprint()` produces:

```json
{
  "preferredVersions": { "php": "8.3", "moodle": "5.0" },
  "steps": [
    {
      "step": "installMoodle",
      "options": {
        "siteName": "PR #42 Preview",
        "adminUser": "admin",
        "adminPass": "password"
      }
    },
    { "step": "login", "username": "admin" },
    {
      "step": "installMoodlePlugin",
      "url": "https://github.com/owner/moodle-mod_example/archive/refs/heads/feature-branch.zip"
    }
  ]
}
```

The plugin ZIP URL is a GitHub archive URL for the PR head branch. Moodle Playground auto-detects plugin type and name from the repository name (e.g., `moodle-mod_example` -> type `mod`, name `example`).

### Preview URL Encoding

Moodle Playground accepts blueprints via URL parameters:

- **Base64 blueprint:** `https://host/?blueprint={base64-encoded-json}` -- used for auto-generated and inline blueprints.
- **Remote blueprint URL:** `https://host/?blueprint-url={encoded-url}` -- used when `blueprint-url` input is provided.

This differs from WordPress Playground which used `data:application/json` URLs.

### Template System

Users can customize the PR description/comment content using templates with `{{VARIABLE}}` syntax. Available variables:

| Variable | Source |
|----------|--------|
| `PLAYGROUND_BUTTON` | Rendered `<a>` tag with preview link |
| `PLAYGROUND_URL` | Full Playground URL with blueprint |
| `PLAYGROUND_BLUEPRINT_JSON` | Raw blueprint JSON string |
| `PLAYGROUND_HOST` | Base Playground URL |
| `PR_NUMBER`, `PR_TITLE`, `PR_HEAD_REF`, `PR_HEAD_SHA`, `PR_BASE_REF` | PR metadata |
| `REPO_OWNER`, `REPO_NAME`, `REPO_FULL_NAME`, `REPO_SLUG` | Repository metadata |
| `PLUGIN_PATH`, `PLUGIN_SLUG` | Plugin input and derived slug |
| `MOODLE_VERSION` | Moodle version input |

All variables are HTML-escaped in the output except `PLAYGROUND_BUTTON` (which contains trusted HTML).

### PR Description Management

In `append-to-description` mode, the action wraps its content in HTML comment markers:

```html
<!-- moodle-playground-preview:start -->
<a href="...">🎓 <strong>Preview in Moodle Playground</strong></a>
<!-- moodle-playground-preview:end -->
```

The action detects and updates this block on subsequent runs. It respects user modifications:

- If the user replaces the content between markers with non-button text, the action treats it as a user placeholder and skips updates.
- If `restore-button-if-removed` is `false` and the user deletes the entire block, it stays deleted.

### Comment Management

In `comment` mode, the action searches PR comments for one containing the hidden `<!-- moodle-playground-preview-comment -->` marker. It uses lazy pagination (100 comments per page, stops at first match) to avoid unnecessary API calls on PRs with many comments.

## Composite Action: expose-artifact-on-public-url

Located at `.github/actions/expose-artifact-on-public-url/action.yml`. This is a **separate, reusable composite action** for advanced use cases where a Moodle plugin needs a CI build step before preview.

### What It Does

1. Downloads a CI build artifact (ZIP) from the current or a different workflow run.
2. Renames it to `pr-{NUMBER}-{SHA}.zip`.
3. Uploads it to a GitHub release (creates a draft release if needed).
4. Outputs a public download URL.
5. Cleans up old artifacts for the same PR (configurable retention).

### When to Use It

Use this when a plugin requires compilation, transpilation, or bundling before it can be installed in Moodle. The pattern is:

1. **Workflow 1** (`pull_request`): Build the plugin ZIP and upload as an artifact.
2. **Workflow 2** (`workflow_run`): Use `expose-artifact-on-public-url` to publish the ZIP, then call this action with a custom blueprint pointing to the public URL.

### Key Inputs

| Input | Required | Default | Purpose |
|-------|----------|---------|---------|
| `artifact-name` | yes | -- | Name matching `upload-artifact` |
| `artifact-filename` | no | `plugin.zip` | ZIP filename inside the artifact |
| `artifact-source-run-id` | no | current run | Workflow run ID that produced the artifact |
| `pr-number` | yes | -- | PR number for artifact naming |
| `commit-sha` | yes | -- | Commit SHA for unique naming |
| `artifacts-to-keep` | no | `2` | Retention count per PR (or `keep-all`) |
| `release-tag` | no | `ci-artifacts` | GitHub release tag to use |
| `github-token` | no | `GITHUB_TOKEN` | Token with `contents: write` |

### Key Outputs

| Output | Format |
|--------|--------|
| `artifact-url` | `https://github.com/OWNER/REPO/releases/download/TAG/pr-NUMBER-SHA.zip` |
| `artifact-name` | `pr-NUMBER-SHA.zip` |

## Relationship to Moodle Playground

[Moodle Playground](https://github.com/ateeducacion/moodle-playground) is a browser-based Moodle LMS running on PHP-WASM (WebAssembly). Key concepts relevant to this action:

- **Blueprints**: JSON files that define the desired state of a Moodle instance (install options, users, courses, plugins). See [blueprint-json.md](https://github.com/ateeducacion/moodle-playground/blob/main/docs/blueprint-json.md).
- **`installMoodlePlugin` step**: Downloads a plugin ZIP from a URL, extracts it to the correct Moodle directory, and runs the Moodle upgrade process. Auto-detects plugin type/name from the repo name.
- **`installTheme` step**: Same as `installMoodlePlugin` but for themes specifically.
- **Blueprint loading**: Via `?blueprint={base64}` or `?blueprint-url={url}` query parameters.
- **Ephemeral**: All state lives in memory (MEMFS). Closing the tab destroys everything. No persistence.
- **Supported Moodle versions**: 4.4, 4.5, 5.0, 5.1, and main.
- **Default credentials**: admin / password.
- **Default host**: `https://ateeducacion.github.io/moodle-playground`.

### Moodle Plugin Naming Convention

Moodle plugins follow the pattern `moodle-{type}_{name}`:

| Repository name | Type | Name | Install path |
|----------------|------|------|-------------|
| `moodle-mod_board` | `mod` | `board` | `mod/board/` |
| `moodle-block_participants` | `block` | `participants` | `blocks/participants/` |
| `moodle-local_staticpage` | `local` | `staticpage` | `local/staticpage/` |
| `moodle-theme_moove` | `theme` | `moove` | `theme/moove/` |

Moodle Playground's `installMoodlePlugin` step auto-detects these from the GitHub archive URL.

## Key Differences from the WordPress Original

| Aspect | WordPress version | Moodle version |
|--------|------------------|----------------|
| Playground host | `playground.wordpress.net` | `ateeducacion.github.io/moodle-playground` |
| Blueprint schema | WordPress Playground Blueprints | Moodle Playground Blueprints |
| Plugin install step | `installPlugin` with `git:directory` resource | `installMoodlePlugin` with ZIP URL |
| Theme handling | Separate `theme-path` input + `installTheme` step | Themes are plugins; use `installMoodlePlugin` or custom blueprint |
| URL encoding | `?blueprint-url=data:application/json,...` (URL-encoded JSON) | `?blueprint={base64}` (base64-encoded JSON) |
| Proxy server | Required `github-proxy.com` for partial directory archives | Not needed; uses full GitHub archive URLs |
| HTML markers | `<!-- wp-playground-preview:start/end -->` | `<!-- moodle-playground-preview:start/end -->` |
| Comment marker | `<!-- wp-playground-preview-comment -->` | `<!-- moodle-playground-preview-comment -->` |
| Brand color | Blue `#3858E9` | Orange `#f98012` |
| Extra inputs | `theme-path` | `moodle-version` |

## Common Tasks for AI Agents

### Adding a new action input

1. Add the input in `action.yml` under `inputs:`.
2. Read it in `src/index.js` with `core.getInput('input-name', {required: false})`.
3. If it should be available in templates, add it to the `templateVariables` object (lines 188-208).
4. Update `README.md` with documentation.
5. Run `npm run build` to regenerate `dist/index.js`.

### Changing the default blueprint

Edit the `buildAutoBlueprint()` function (line 98). The function returns a JSON string containing Moodle Playground blueprint steps. See the Moodle Playground blueprint documentation for available step types.

### Changing the preview button appearance

Edit the `defaultButtonTemplate` string (line 172). This is an HTML template with `{{PLAYGROUND_URL}}` for the link target. The SVG asset at `assets/playground-preview-button.svg` is available for image-based buttons but is not currently used in the default template.

### Supporting built artifacts (plugins with a build step)

Use the two-workflow pattern:

1. Workflow 1 builds the plugin and uploads the ZIP with `actions/upload-artifact`.
2. Workflow 2 uses `expose-artifact-on-public-url` to publish the ZIP to a release, then calls this action with a custom `blueprint` input containing an `installMoodlePlugin` step pointing to the public URL.

### Testing locally

There is no test suite. To verify changes:

1. Edit `src/index.js`.
2. Run `npm run build`.
3. Push to a branch and create a test PR on a repository that uses this action.
4. Check the PR description/comments for the preview button.

### Publishing a new version

1. Ensure `dist/index.js` is up to date (`npm run build`).
2. Commit all changes.
3. Create a GitHub release with a semver tag (e.g., `v1.0.0`).
4. Update the major version tag: `git tag -fa v1 -m "Update v1 tag" && git push origin v1 --force`.
5. The `publish-immutable-action.yml` workflow handles marketplace publishing automatically.
